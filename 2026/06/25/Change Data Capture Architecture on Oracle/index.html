Change Data Capture Architecture on Oracle

2026-06-25

1. Oracle Architecture

1. Cluster & Oracle Grid Infrastructure(GI)

Cluster is a collection of interconnected physical servers or virtual mathines(nodes) operating as a single system. A single-node environment is a single instance system.
Grid Infrastructure (GI): software framework running on the nodes that provides:

  • Oracle Clusterware: manages cluster node membership, monitors server health and provides high availablitity
  • Oracle ASM(automatic storage management): a specialized volume manager and file system optimized specifically for oracle database files, striping and mirroring data across shared storage disks.
  • SCAN Listerner(Single Client access name): a cluster-level virtual network entity taht acts as the single entry point for all incoming remote client connections, routing traffic across the nodes.

2. Oracle Instance & memory structures

System Global Area(SGA) is a massive, shared memory allocated during the instance bootstrap phase. It is universally addressable by all background and forground processes running onthe node OS. including SHared pool, buffer cache, redo log buffer, large pool.

Program Global Area(PGA): private memory for a query session, total memory used by the PGA grows and shrinks dynamically as users log in and log out.

  • allocates isolated runtime memory to process CPU-heavy execution operators like hash joins, index builds, and data sorting algorithms.
  • LogMiner Session Registry Memory: A dedicated sub-allocation track within the session’s PGA context . it records runtime configurations for log parsing operations, active SCN limits, registered file handles, and dictionary parsing structures. Beacuse it is bound to the connection’s PGA, a LogMiner session is intrinscically linked to the lifetime of the client connection thread.

Dynamic Performance Views (V$) are real-time instrumentation interfaces that read directly from volatile database kernel memory maps inside the SGA and PGA. These views are virtual structures that do not read traditional disk block tablespaces and clear out entirely when the database instance shuts down.

  • V$LOG: Queries the log group tracking control structures in the SGA. It exposes the current log sequence number, group identifiers, block allocation metrics, and state conditions (CURRENT, ACTIVE, INACTIVE, UNUSED).

  • V$LOGFILE: Evaluates the multiplexing configurations of the transaction log architecture, mapping each database logical log group number to its literal operating system file system string path.

  • V$LOGMNR_SESSION: Exposes global metadata properties for all active mining sessions initialized across the instance, detailing memory overhead metrics inside individual client PGAs.

  • V$LOGMNR_CONTENTS: The primary relational view used to extract processed transaction entries. Querying this view forces LogMiner to parse registered binary logs on the fly, transforming raw hexadecimal change vectors into human-readable text strings within the output rows (SQL_REDO and SQL_UNDO).

  • V$LOGMNR_LOGS: Enumerates every physical log or archive file registered for processing in the active session context, showing their individual SCN boundary limits.

  • V$LOGMNR_PARAMETERS: Tracks active operation parameters, transformation options, and output filter flags passed to the session during engine initialization.

  • workload processing: provide CPU cycles and memory to handle sql parsing, analysis, optimization/planning, query execution, and the execution of stored procedures/functions

  • One node is a physical or logical server harware and OS (linux or windows), one or more instances run on a node

  • Connection paradigms:

    • Local administration: SQL*Plus on server via IPC or Bequeath protocol, allowing authentication via host OS privileges (SYSDBA/SYSOPER) without requiring a network listener
    • Remote Clients: External applications (dbeaver or applications) connect over TCP/IP via the Oracle Net Listener(or SCAN listener in RAC environments)

3. Database (physical files on shared storage)

Oralce’s database strictly refers to the physical files sitting on the disk storage (Datafiles, Control Files, Redo Logs and Tempfiles)

  • Single instance setup: a single instance mounts and opens a single databse.
  • Oracle RAC setup: multiple independent instances running on distinct cluster nodes simultaneously mount and open the exact same set of physical database files on shared storage, using Cache Fusion to coordinate memory and locking across the cluster interconnect.

4. Logical Abstraction(Multitenant Architecture)

CDB(Container database)

CDB is the overarching root container(CDB$ROOT) that houses the master data dictionary, background metadata and common users(system accounts or users prefixed with C##).
Core administrative dictionary tables
- OBJ$: Automatically assigns a unique binary internal object identifier number to every schema entity (table, view, sequence, trigger) instantiated inside the database.
- USER$: Manages base user profiles, authentication parameters, cryptographic password hashes, and security clearance roles.
- TAB$: Dictates table layout configurations, mapping specific table identifiers directly to physical storage block groupings, clustering definitions, and column metadata.

PDBs(pluggable databases)

PDBs are fully isolated user-facing logical databases nested in CDB, contain application schemas, local tables, and Local users whose credentials exist exclusively inside that specific PDB dictionary.

2. Oracle Redo Log Mechanics

Buffer Cache & Log Buffer

when a DML was rquested from client, the database executes two parallel, independent operations in memory:

  • Buffer Cache / DBWn (Data Path):
    The database modifies the data block directly within the SGA buffer cache, transitioning it to a “dirty buffer”. The DBWn lazily batches and flushes these dirty buffers to the physical datafiles.

  • Log Buffer / LGWR (Redo Path):
    Concurrently, a redo record is constructed within the log buffer.
    Redo records are complex logical structures, generally capped at an upper size limit of 65,536 bytes.
    Each redo record is prepended with a header containing a VLD field that dictates the record type.

Only after the LGWR has flused the corresponding redo record to disk (Current Online Redo Log), DBWn can writie a dirty data block to a datafile.

Once a COMMIT statement comes, the session enter a synchronous wait state, waiting for the LOG_FILE_SYNC event.
Before this event trigger, LGWR will aggressively flushes the log buffer to the CURRENT online redo log files.
The COMMIT is acknowledged to the application only once LGWR confirms the redo record is durably written, entirely ignoring DBWn’s lazy datafile flush.

Redo Record

By default, Oracle redo logs are designed exclusively for instance recovery (roll-forward operations). They record physical changes at the block and byte-offset level.

Redo Block & Redo Record
Redo Block is the minimum fundamental unit of physical input/output (I/O) for the redo log files.
Standard Oracle architecture typically defines a redo block size of 512 bytes (though 4KB is utilized on specific modern storage hardware).
Redo Record is a logical data structure comprising one or more change vectors that precisely document the physical modifications applied to database data blocks.
The relationship between blocks and records is entirely dynamic: a single physical redo block may encapsulate multiple small redo records, or a single large redo record may sequentially span multiple physical redo blocks.

SCN Assignment
The primary granularity of SCN assignment is the transaction commit.

When an UPDATE executes, the default redo record contains only

  • the physical ROWID
  • the specific column values that were modified
    Without supplemental logging, LogMiner will extract the chagne, but the resulting SQL WHERE clause will reply entirely on the physical ROWID.
    It will not contain the logical identifier(the primary key) unless the primary key itself was the column being updated.
    Because ROWIDs are physical pointers that can change across database instances or during maintenance operations(like table moves), ROWID-based SQL is useless for logical replication to another database(such as PostgreSQL) or for precise row-level auditing.

Supplemental Logging

To facilitate logical CDC, Supplemental Logging must be explicitly enabled.
For UPDATE and DELETE operations, the database logs the pre-image of the row’s primary key or unique index to allow the replication engine to formulate a deterministic WHERE clause.
If a table lacks a primary key, Identification Key Logging must be configured to log ALL COLUMNS, which massively inflates the redo generation volume.

Supplemental logging instructs the database to write additional data columns into the redo stream to fully qualify a row logically. Beyond the primary key and unique index pre-images, it can include several other components depending on the configuration level (Ddatabase-level or Table level);

  1. Minimum Supplemental Logging
    This is the baseline requirement for LogMiner. It adds minimal information necessary to identify row structures, specifically handling:

    • Chained Rows: information to correctly reconstruct rows that span multiple data blocks
    • Storage Arrangements?: Metadata to interpret cluster tables or indexed-organized tables (IOTs)

  2. Identification Key Logging
    When enabled at the database or table level, this adds specific constraints to the redo stream for every UPDATE operation:

    • Primary Key Logging: The pre-image of all columns comprising the primary key.
    • Unique Index Logging: The pre-image of unique index columns(typically leveraged when a table lacks a defined primary key).
    • Foreign Key Logging: The pre-image of foreign key columns. This is critical for replication environments to maintain referential integrity and ensure parent-child constraints are evaluated correctly in the target database.
    • All Column Logging: The pre-image of all fixed-lenght columns in the row, regardless of whether they were modified. This is generally reserved for tables without any primary or unique keys, though it generates significant redo volume.

  3. User-Defined Column Groups(Table-Level)
    Administrators can configure supplemental logging to record specific, user-defined groups of columns.

    • Unconditional Supplemental Logging: The database logs the pre-images of specified column group whenever the row is updated, even if none of the columns in the group were changed.
    • Conditional Supplemental Logging: The database logs the pre-images of all columns in the specified group only if at least one column within that specific group is modified.

Multiplexing: Log Groups & Members

Oracle utilizes a specific hierarchical structure for redo logs:

  • Log Group: A logical slot in the circular round-robin buffer, identified by GROUP# and SEQUENCE#.
  • Member: A physical file path. A group comprises one or more identical physical members
    • /u01/redo01a.log
    • /u02/redo01b.log

During a flush, LGWR issues parallel write operations to all available members of the current group simultaneously to guarantee redundancy.
A CDC reader safely ignores multiplexing complexities, requiring only one valid, uncorrupted member file to extract the required SCN sequence.

Checkpoint and Log Lifecycle

  • CDC Reader Checkpoint(Watermark)
    A stateful marker persisted entirely by the external CDC application (or internal capture metadata).
    (It represents the lowest SCN of an unacknowledged transaction that the CDC reader guarantees it can resume from without data loss.)

  • Instance Checkpoint(CKPT)
    Managed by the Oracle instance. It tracks how far DBWn has progressed in flushing dirty buffers to datafiles.
    This SCN is recorded in the controlfile and governs when a redo log is no longer needed for instance crash recovery.
    (the SCN after this CKPT in Archived Log is safe to delete by RMAN.)

The instance checkpoint governs the operational statuses of redo log groups, defining the boundary conditions for external CDC readers:

Log Status Architectural Meaning CDC Reader Implication
CURRENT LGWR is actively appending redo records to the members of this group Provides the lowest-latency read path, but is highly volatile and subject to overwriting if the status change
ACTIVE The group has undergone a log switch, closing it to new writes, but the CKPT SCN has not yet advanced past all changes it protects. Physically immutable to LGWR in this state, but remains at risk of transitioning to INACTIVE
INACTIVE The CKPT process confirms all dirty buffers protected by this group are safely written to datafiles Extremely volatile; eligible for immediate resue by LGWR’s round-robin cycle if archiving is complete.
UNUSED The group has never been written to since creation Ignored by CDC readers

This status progression underscores a critical engineering vulnerability for external CDC implementations: attempting to read from a CURRENT or ACTIVE online redo log is inherently racy.
If a reader is parsing an online log and the database instance subsequently marks the group as INACTIVE, the round-robin cycle may allow LGWR to overwrite the physical file with a new sequence number mid-read, resulting in torn blocks.
Robust readers detect this race condition via coarse-grained block-header and file-header validations, instantly aborting the torn read and falling back to the immutable archived copy.

LogMiner(DBMS_LOGMNR)

Oracle LogMiner serves as the foundational decoding engine capable of parsing the raw binary redo byte streams into structured LCRs and exposing them via the V$LOGMNR_CONTENTS view.

The deprecation of CONTINUOUS_MINE. In legacy Oracle versions, a CDC application could invoke the DBMS_LOGMNR.START_LOGMNR procedure utilizing the CONTINUOUS_MINE constant, which instructed LogMiner to automatically discover log sequences and seamlessly cross file boundaries in real-time. Oracle officially deprecated CONTINUOUS_MINE in 12c Release2(12.2) and completely desupported it in 19c and subsequent releases. While Oracle’s official documentation explicitly states there is no relacement functionality and does not publicly publish internal failure mechanisms in the desupport notice, architectural consensus attributes its removal to the inherent fragility and overwrite-race conditions of attempting to perform continuous validation against actively written online redo logs without internal lock coordination.

3. Oracle CDC Reader (GoldenGate, Debezium)

1. LogMiner server(MSnn processes)

Oracle embed the LogMining server directly into the Oracle database engine as a suite of background processes.
The collective assembly of the Reader, Preparers, and Builder is officially designated by Oracle as the LogMiner server.

  • Reader Process: Interfaces with the log buffer or files to intercept and split redo vectors. Initial physical I/O and splitting the redo vectors into smaller, manageable chunks.

    LogMiner server utilizes a tiered ingestion hierarchy to bypass the torn-read race conditions of online files entirely:
    1. SGA Memory(Primary): the engine reads redo directly from the log buffer in the SGA in real-time, operating in lockstemp with LGWR before data hits the disk.
    2. Online Redo(Fallback): If the capture process lags slightly behind memory retention, it reads from the intact online redo logs.
    3. Archived Redo(Deep Fallback): If the process falls significantly behind, it falls back to the immutable archived logs.

  • Preparer Process(es): Scans the memory regions, pre-filters the data against capture rules, and evaluate the vectors against the configured capture rules, filter out irrelevant data, and decode the binary redo stream into logical tuples (representing the actual DML or DDL operations)

  • Builder Process: Ingests the disjointed records from the Preparers and rigorously merges them into strict SCN chronological order

  • Capture Process: Finalizes the standard LCRs and routes them to the outbound queue.

LogMiner server is the built-in Oracle proecesses to extract the Redo logs, which be utilized by LogMiner package, Integrated Capture of GoldenGate, and the XStream Out.
LogMiner server is responsible for reading the physical redo vectors fromt he log buffer or files, filtering them against capture rules, and reconstructing the physical changes into chronological Logical Change Records (LCRs).

2. Integrated Capture & XStream Out

Integrated Capture is the CDC reader of GoldenGate.
XStream Out is a Native OCI Application Programming Interface API built-in oracle to expose the power of Integrated Capture to enterprise C/C++ applications.
Both need need Oracle GoldenGate license.

Integrated Capture & XStream Out

  • Base Layer: LogMiner Server
  • Middle Layer: Outbound Server
    This is a memory-based queue structure localted within the database SGA. It receives the rigorously ordered LCR stream from the LogMiner server’s builder process and manages the state and transmission queuing for external consumers.
  • Top Layer: Parallel CLient Interfaces
    This layer represents the external consumption of the LCR stream. This is where the architecture strictly diverges into two independent, parallel pathways. They sit side-by-side atop the Outbound Server:
    • Endpoint A: Integrated Capture. The Oracle GoldenGate Extract process connects directly to the Outbound Server to receive the LCRs for propagation through the proprietary GoldenGate replication topology.
    • Endpoint B: XStream OUT. This expose a native C-level Oralce Calll Interface (OCI) or Java API. It connects directly to the Outbound Server, entirely bypassing Integrated Capture, to push LCRs over a network socket to custom compiled client binaries.

Streams Pool Sizing and MAX_SGA-SIZE Optimization
Integrated Capture dynamically allocates memory exclusively from the Streams Pool portion of the SGA.
The MAX_SGA_SIZE parameter caps the volume of shared memory the server can request.
If MAX_SGA_SIZE is undersized, the engine exhausts its allocation and violently abotrts, or aggressively spills to disk-backed temporary segments, spiking latency.

Low-Water Mark (LWM)
A resilient CDC system must periodically notify the outbound server of its durably processed position via OCIXStreamOutProcessedLWMSet.
The database utilizes the OCI_ATTR_XSTREAM_ACK_INTERVAL attribute (default 30s) to forcefully preempt OCIXStreamOutLCRReceive loops, allowing the application thread to transmit the acknowledgment packet safely.
This purges synchronized transactions from the SGA and releases the underlying dependency on associated archive logs.

3. Third-Party CDC (Debezium)

For massive, distributed streaming architectures standardizing on Apache Kafka, the open-source Debezium Oracle Connnector is the primary conduit.
LogMiner Package is an PLSQL interface of the LogMiner server.
The absence of COTINUOUS_MINE mandates that modern external CDC readers implement a deterministic, stateful driver loop - a “safe subset” of continuous mining - to manage log file resolution and mining boundaries:

  1. Resolution via Range Scanning: The application queries the V$ARCHIVED_LOG and V$LOG views to resolve the physical file paths encompassing the target SCN range. The logic executes a range scan against the FIST_CHANGE# and NEXT_CHANGE# columns.
  2. Explicit File Registration: The resolved files are registered with the mining session using DBMS_LOGMNR.ADD_LOGFILE with the DBMS_LOGMNR.NEW or DBMS_LOGMNR.ADDFILE options.
  3. Execution Bounding: The mining window is activated via DBMS_LOGMNR.START_LOGMNR, passing explicit STARTSCN and ENDSCN parameters, combined with DBMS_LOGMNR.DICT_FROM_ONLINE_CATALOG and COMMITTED_DATA_ONLY flags.
  4. Data Consumption: The client streams decoded records by polling V$LOGMNR_CONTENTS. For CDB architectures, queries filter the CON_ID column to isolate changes for sepcific PDBs.
  5. Session Termination: The session must be cleanly terminated via DBMS_LOGMNR.END_LOGMNR to release PGA memory. The internal checkpoint advances, and the application orchestrates the jump to the next sequence.

Advanced Transaction Buffering
To guarantee atomic, COMMITTED_DATA_ONLY delivery, Debezium must intercept every event for an active Transaction ID(XID), buffer the payload indefinitely, and suspend transmission until an explicit COMMIT or ROLLBACK is observed.
Pathological batch transactions cause the default JVM memory buffer (log.mining.buffer.type = memory) to trigger severe garbage collection thrashing and OutOfMemoryError failures.
Modern deployments transition to off-heap memory buffering (infinispan_embedded).
This hybrid architecture seamlessly spills data from physical memory to persistent disk storage (persistence) when the memory capacity threshold (max-count) is breached.
This ensures sub-millisecond memory latencies for OLTP traffic while providing unbounded disk durability for massive batch updates without crashing the Kafka Connect worker.

Retention Coordination and RMAN Policy Failures
The ultimate reliability of any redo-based CDC pipeline is constrained by the physical lifecycle of the archived redo logs on disk.
If the consumer pipeline stalls, the CDC reader’s required checkpoint SCN freezes, but the Oracle database mercilessly continues to generate and archive new logs.

The Missing Log Incident (ORA-01291) and Re-instantiation
Oracle Recovery Manager (RMAN) manages archive deletion independent of external CDC readers.
Standard maintenance procedures execute aggressive cleanups (e.g., DELETE ALL INPUT, which purges from all archive destinations, versus DELETE INPUT, which purges only from the backed-up destination).
If RMAN deletes a log containing the CDC reader’s required checkpoint SCN, the pipeline is irreparably severed, resulting in an ORA-01291: missing log file error.
Recovery requires a disruptive full re-instantiation: administrators must select a consistent SCN in the present, execute a bulk table unload using Oracle Flashback query syntax (SELECT * FROM table AS OF SCN :s), load the data into the target, and reset the CDC reader’s checkpoint to begin streaming strictly after the snapshot SCN.

Enforcing Strict Retention with Dual-Guard Deletion Policies
To eradicate ORA-01291 incidents, administrators must bind the RMAN deletion policy to the replication pipeline’s progress.
For Integrated Capture (GoldenGate), the extract process registers directly with the database, and RMAN natively honors the REQUIRED_CHECKPOINT_SCN, autonomously refusing to purge needed logs.
For external readers like Debezium, administrators forge a protective RMAN policy utilizing dual-guard directives :

By enforcing APPLIED ON ALL STANDBY alongside a BACKED UP directive, the policy guarantees that no archived redo log is subjected to deletion until the file has been both safely backed up to tertiary storage and explicitly consumed by the configured CDC endpoints, mathematically prioritizing the chronological consistency of the replication stream over raw storage utilization.

4. RAC orchestration and SCN Synchronization

In a RAC rchitecture, multiple instances(nodes) share the same datafiles, but each instance maintains its own private thread of redo logs and distinct sequence numbers.
A global application transaction modifies data across multiple nodes, scattering its composite redo fragments across physically disjointed threads.
A CDC reader must perform a rigorous k-way multi-thread heap merge, interleaving records from all threads simultaneously to reconstruct a globally consistent stream.

SCN/Sequence Number
In an Oracle database, there is strictly one global Sytem Change Number(SCN) namespace that totally orders all changes across the entire cluster.
Sequence numbers are per-thread(e.g., Thread 1 Sequence 615 and Thread 2 Sequence 615 are completely different physical files), but the SCN is universal.

Broadcast-on-Commit(BOC)
To execute an accurate multi-thread merge, the SCN must function as a perfectly synchronized global logical clock.
Modern architectures utilize the Broadcast-on-Commit (BOC) protocol (MAX_COMMIT_PROPAGATION_DELAY = 0).
Under BOC, when an instance commits, LGWR synchronously broadcasts the updated SCN to the Lock Management Server (LMS) processes on all peer nodes.
On modern Exadata infrastructure (21c+), this is optimized via BOC over RDMA, bypassing LMS processes and utilizing atomic RDMA hardware operations to directly overwrite SCN memory registers in remote SGAs, eliminating log write broadcast wait network overhead.

Resolving the Idle Thread Low-Watermark Stall
The multi-thread merge algorithm contains a critical logical flaw: the global low-watermark SCN cannot advance past the lowest confirmed position across all active RAC threads.
If Thread 3 is idle while Threads 1 and 2 advance rapidly, the reader cannot safely emit records from Threads 1 and 2, as it cannot mathematically guarantee Thread 3 will not suddenly emit a delayed record at an older SCN.
Mature CDC architectures eradicate this stall condition by injecting synthetic heartbeats.
A scheduler executes a superficial UPDATE on a dummy table across every RAC instance at a rigorous interval, forcing a continuous trickle of redo generation across all physical threads.
This proves to the CDC reader that the idle node’s SCN is advancing, unlocking the k-way merge low-watermark.

CDC Terminology

  • CDC/DML/DDL

  • LSN/SCN/GTID

    • Log Sequence Number
    • System Change Number
    • Global Transaction ID

  • Snapshot (Initial load/ bootstrap)
    A full consistent copy of existing table data taken before streaming incremental changes, so the target starts from a complete baseline.

  • Watermark
    The recorded position(LSN/SCN/offset) up to which changes have been successfully captured/committed. Enables resume after restart.

  • Tombstone
    A marker representing a delete - a record whose presence signals “this key no longer exists,” important for downstream compaction.

  • Purpost and mechanism

    1. Signaling deletion in append-only logs
      Many CDC systems and message brokers (e.g., debezium, Apache Kafka) utilize append-only commit logs to stream events. Because you cannot modify or delete past messages in an append-only log, a deletion cannot be represented by simply removing a message. Instead, the CDC engine appends a new record-the tombstone-to the log.
    2. Structure of a Tombstone Record
      a. The primay key of the deteted row (to identify which record to remove)
      b. A apyload/value set to null (or a specific delete operation flag, such as “op”:”d”)
      c. Log compaction enablement, In distributed streaming platforms like Kafka, topic partitions undergo log compaction to conserve storage. Compaction retains only the latest message for each primary key.
      Tombsone is critical in Database Migration
      During a zero-downtime database migration, the migration pipleline must maintain exact parity between the source and target systems.
      Without tombstone records, a CDC-driven migration cannot safely handle hard deletes, rendering true transactional synchronization impossible.

  • High-water mark (HWM)
    The boundry between the snapshot phase and the streaming phase; changes after it are read from the log, before it from the snapshot.

  • Change Records

    • Before/after Image (Pre/post image)
      The row’s column values before and after a change. Updates need both; the before-image is essential for deletes and for matching keys.

    • Op type or Operation type
      Flag on each change event: c/I(craete/insert), u/U(update), d/D(delete), r(read/snapshot)

    • Idempotency (Delivery guarantees)
      Property that re-applying the same change event yields the same result - critical because CDC delivery is usually at-least-once

      • At-least-once: may duplicate events on retry (needs idempotent apply)
      • Exactly-once: is stronger but harder/costlier.

    • Commit order
      Events must be applied in source transaction-commit order to preserve consistency, especially across related rows.

    • Transaction boundaries (BEGIN/COMMIT framing)
      Grouping change events by the transaction they belonged to, so consumers can apply atomically.

    • WAL (Write-ahead log)
    • Logical replication / logical decoding
    • Replication slot
